# Исследование: Telegram Mini App для бьюти-мастера

## Цель проекта

Каталог услуг бьюти-мастера как Telegram Mini App: клиент открывает ТГ-бота → нажимает кнопку → видит услуги с ценами и кнопкой «Выбрать».

---

## 5 примеров Telegram Mini App в бьюти и смежных нишах

### 1. YCLIENTS / Dikidi через Telegram Mini App

**Что делает:** Крупные SaaS-платформы онлайн-записи предоставляют интеграцию через Telegram-ботов с Mini App. Клиент нажимает кнопку в боте и попадает в веб-приложение записи прямо внутри Telegram.

**Сильные стороны:**
- Полноценный флоу записи: услуга → мастер → дата/время → подтверждение
- Синхронизация с CRM салона
- Напоминания о записи через бот
- Онлайн-оплата

**Слабые стороны:**
- Шаблонный дизайн без индивидуальности мастера
- Нет портфолио (фото работ) — сразу бросает в запись
- Интерфейс перегружен, если много услуг
- Нет витрины/каталога — только утилитарная запись

**Что мы можем сделать лучше:**
- Добавить визуальную витрину с фото работ до записи
- Сделать персонализированный дизайн под конкретного мастера
- Упростить интерфейс — минимум экранов, максимум наглядности

---

### 2. Кастомные Mini App от индивидуальных мастеров (nail-мастера, бровисты)

**Что делает:** Частные мастера заказывают или сами делают простое Mini App как визитку + запись. Бот — точка входа, Mini App — каталог работ и форма записи.

**Сильные стороны:**
- Персонализированный дизайн, акцент на визуал (фото работ)
- Простота — 3-4 экрана максимум
- Быстрая загрузка
- Ощущение «личного приложения мастера»

**Слабые стороны:**
- Нет реальной интеграции с календарём (запись через «напишите мне»)
- Нет онлайн-оплаты
- Нет отзывов клиентов
- При обновлении портфолио нужно менять код

**Что мы можем сделать лучше:**
- Сохранить простоту и визуальность, но добавить реальную кнопку записи
- Дать мастеру админку для обновления услуг без разработчика
- Добавить секцию отзывов

---

### 3. Конструкторы Mini App (BotMother, SendPulse, Smartbot Pro)

**Что делает:** Платформы-конструкторы позволяют создать Mini App без кода. Много салонов используют их для каталогов услуг с записью.

**Сильные стороны:**
- Быстрый запуск, не нужен разработчик
- Есть шаблоны под бьюти-нишу
- Встроенная аналитика

**Слабые стороны:**
- Ограниченная кастомизация дизайна
- Выглядит как «интернет-магазин», а не бьюти-сервис
- Нет анимаций и премиального ощущения
- Ограниченные возможности по интерактивному выбору времени

**Что мы можем сделать лучше:**
- Создать premium-ощущение: плавные анимации, haptic feedback
- Дизайн, заточенный именно под бьюти (визуал, цвета, типографика)
- Нативное поведение внутри Telegram (MainButton, BackButton, themeParams)

---

### 4. @DurgerKingBot (Durger King) — демо-каталог от Telegram

**Что делает:** Официальный showcase от команды Telegram. Имитирует приложение для заказа еды. Пользователь нажимает «Menu» — открывается Mini App с каталогом блюд, выбором количества, корзиной.

**Сильные стороны:**
- Нативное ощущение — плавные анимации, адаптация под тему Telegram
- Использует MainButton внизу экрана — «VIEW ORDER (сумма)»
- Очень быстрая загрузка, минимализм
- Haptic feedback при действиях
- Открытый исходный код — можно изучить как reference

**Слабые стороны:**
- Нет поиска по меню
- Нет описания товаров — только название и цена
- Нет фильтров
- Нет истории заказов

**Что мы можем сделать лучше:**
- Взять за основу UX-паттерн (карточки → корзина → MainButton), но добавить описания услуг, длительность, фото
- Добавить категории услуг

---

### 5. Боты доставки еды (локальные рестораны и кафе через Mini App)

**Что делает:** Локальные рестораны запускают Mini App с полным меню, выбором блюд, корзиной и оформлением доставки. Паттерн аналогичен каталогу услуг.

**Сильные стороны:**
- Горизонтальные табы категорий — привычный и удобный паттерн
- Промокоды и акции в интерфейсе
- Push-уведомления через Telegram (не нужно отдельное приложение)
- Нет необходимости устанавливать отдельное приложение

**Слабые стороны:**
- Часто медленная загрузка (тяжёлые изображения)
- Теряется состояние при свайпе назад (закрывается Mini App целиком)
- Неудобная навигация «назад» — нет встроенного роутинга
- Оплата вынесена за пределы Mini App
- Не сохраняются предпочтения между сессиями

**Что мы можем сделать лучше:**
- Оптимизировать загрузку изображений (lazy loading, WebP, placeholder)
- Реализовать внутренний роутинг с поддержкой BackButton Telegram
- Сохранять состояние пользователя через Telegram CloudStorage API

## Обязательные элементы UI

### Telegram-нативные
- **MainButton** — основная кнопка внизу экрана (например, «Записаться — 2500 ₽»)
- **BackButton** — нативная кнопка «назад» от Telegram
- **themeParams** — адаптация цветов под светлую/тёмную тему пользователя
- **HapticFeedback** — вибрация при нажатиях (ощущение нативности)
- **CloudStorage** — сохранение предпочтений между сессиями

### UX-элементы
- **Карточки услуг** — фото + название + цена + длительность + кнопка действия
- **Категории** — горизонтальные табы или фильтр для навигации по типам услуг
- **Быстрая загрузка** — критично, пользователь ожидает мгновенного открытия
- **Skeleton-загрузка** — скелетоны вместо спиннеров при загрузке данных
- **Pull-to-refresh** — привычный жест обновления

## Ключевые выводы для нашего проекта

### Конкурентные преимущества, которые стоит реализовать

1. **Визуальность** — в отличие от YCLIENTS/Dikidi, делаем акцент на фото работ мастера
2. **Простота** — 2-3 экрана для MVP (витрина → услуга → действие), без перегруза
3. **Нативность** — использовать все возможности Telegram API (MainButton, haptic, тема)
4. **Персонализация** — дизайн, который ощущается как «приложение мастера», а не шаблон
5. **Скорость** — мгновенная загрузка, оптимизированные изображения

### MVP-экраны (минимум для запуска)

1. **Каталог услуг** — карточки: фото, название, цена, длительность, кнопка «Выбрать»
2. **Детали услуги** — описание, фото, кнопка «Записаться» (MainButton)
3. **Действие** — переход к записи через сообщение боту (MVP) или встроенная форма (v2)

### Технические рекомендации

- Использовать `Telegram.WebApp.themeParams` для адаптации цветов
- Использовать `Telegram.WebApp.MainButton` для основного CTA
- Использовать `Telegram.WebApp.BackButton` для навигации
- Использовать `Telegram.WebApp.HapticFeedback` для тактильной отдачи
- Оптимизировать изображения (WebP, lazy loading)
- Минимизировать количество запросов к серверу
- Обеспечить работу без интернет-задержек (кэширование, оптимистичный UI)

## Экспертная оценка

### Взгляд UX-дизайнера

**Типичные ошибки новичков:**

1. **Слишком много экранов до целевого действия.** Клиент хочет увидеть услуги и записаться. Каждый лишний тап — потеря 20-30% пользователей. Если до кнопки «Записаться» больше 2 нажатий — это провал.
2. **Игнорирование темы Telegram.** Mini App, который не подхватывает светлую/тёмную тему, выглядит чужеродно. Пользователь чувствует, что «перешёл на сайт» — доверие падает.
3. **Мелкие кнопки и зоны нажатия.** Минимальный размер тап-зоны — 44×44px. В бьюти-нише клиенты часто с длинными ногтями — кнопки должны быть крупными.
4. **Нет визуальной иерархии.** Все услуги одинакового размера, без акцентов. Глаз не знает, за что зацепиться. Нужно выделять популярные/рекомендуемые услуги.
5. **Отсутствие обратной связи.** Нажал кнопку — ничего не произошло. Нужен haptic feedback, анимация нажатия, изменение состояния кнопки.
6. **Перегруз информацией на карточке.** На карточке в каталоге достаточно: фото, название, цена, длительность. Подробное описание — на экране деталей.

**Что обязательно должно быть:**

- Цена видна сразу на карточке, без «уточняйте»
- Длительность процедуры — клиент планирует время
- Одно главное действие на экран (MainButton)
- Возможность вернуться назад без потери состояния
- Состояния загрузки (skeleton, не спиннер)
- Пустые состояния с понятным текстом (если нет услуг в категории)

**Что бесит пользователей:**

- Долгая загрузка (больше 1-2 сек — уходят)
- «Цена по запросу» / «от 1000 ₽» без конкретики
- Закрытие Mini App при свайпе вниз (случайный жест)
- Невозможность быстро вернуться к каталогу после просмотра услуги
- Формы с кучей обязательных полей для простой записи
- Отсутствие фото работ — «покупаю кота в мешке»

---

### Взгляд маркетолога

**Что заставит записаться (триггеры конверсии):**

1. **Фото работ до/после.** Это главный продающий элемент в бьюти. Клиент хочет видеть результат на реальных людях, а не описание. Фото конвертят в 3-5 раз лучше, чем текст.
2. **Конкретная цена + длительность.** Убирают страх «а вдруг дорого» и «а сколько это займёт». Прозрачность = доверие.
3. **Социальное доказательство.** Количество выполненных процедур, отзывы, рейтинг. Даже простое «Записалось 48 клиентов» работает.
4. **Минимум трения при записи.** Telegram уже знает имя и username клиента. Не нужно просить заполнять формы — передавать данные автоматически через `initData`.
5. **Якорная цена и спецпредложения.** «~~3000~~ 2500 ₽ — первый визит» — классический приём, работает и в Mini App.
6. **Срочность.** «Осталось 2 слота на эту неделю» — мотивирует не откладывать.

**Элементы, повышающие конверсию:**

| Элемент | Влияние на конверсию |
|---|---|
| Фото работ (до/после) | +200-400% |
| Видимая цена на карточке | +50-80% |
| Кнопка «Записаться» на каждой карточке | +30-50% |
| Отзывы/рейтинг | +20-40% |
| «Первый визит — скидка» | +40-60% |
| Указание длительности | +15-25% |
| Меньше полей в форме записи | +20-30% за каждое убранное поле |

**Воронка для MVP:**
```
Открыл бота (100%)
  → Нажал «Посмотреть услуги» (70-80%)
    → Просмотрел каталог (60-70%)
      → Нажал «Записаться» на услуге (15-25%)
        → Отправил заявку боту (10-20%)
```

**Рекомендации:**
- Первый экран должен «продавать» — лучшие фото работ сверху, не логотип
- Кнопка записи должна быть доступна всегда (MainButton), а не в конце скролла
- После записи — предложить «Поделиться с подругой» (виральность через Telegram)
- Бот должен напоминать о записи за 24ч и 2ч — это снижает неявки на 30-50%

### Взгляд Telegram-разработчика

**Ограничения платформы:**

1. **Размер Mini App.** Максимальная высота — зависит от устройства. На маленьких экранах (~568px) контент обрезается. Проектировать под минимум 320×568px.
2. **Закрытие свайпом вниз.** Пользователь может случайно закрыть Mini App жестом. Использовать `Telegram.WebApp.enableClosingConfirmation()` если есть несохранённые данные.
3. **Нет полноценного роутера.** При нажатии системной «назад» (Android) или свайпе — Mini App закрывается. Нужно перехватывать BackButton через `Telegram.WebApp.BackButton` и управлять навигацией вручную.
4. **Ограничения iframe.** Mini App работает в WebView. Нет доступа к камере/геолокации без явного запроса. Нельзя открыть другое приложение напрямую.
5. **initData и авторизация.** Telegram передаёт данные пользователя (id, имя, фото) через `initData`. Эти данные нужно валидировать на сервере через HMAC-SHA256 с bot token. Никогда не доверять клиентским данным без проверки.
6. **CloudStorage лимиты.** Максимум 1024 ключа, значение до 4096 байт каждое. Хватит для настроек, но не для кэширования данных.

**Что работает хорошо:**

- **MainButton** — нативная кнопка внизу экрана. Всегда видна, не скроллится. Идеальна для «Записаться». Поддерживает текст, цвет, состояние загрузки, прогресс.
- **HapticFeedback** — `impactOccurred('light')` при нажатии на карточку, `notificationOccurred('success')` при записи. Мелочь, которая даёт ощущение нативного приложения.
- **themeParams** — автоматически получаем цвета темы пользователя. Используем `var(--tg-theme-bg-color)`, `var(--tg-theme-text-color)` и т.д. через CSS-переменные.
- **sendData / передача данных боту** — при записи отправляем данные обратно в бот через `Telegram.WebApp.sendData()`. Бот получает сообщение с данными и подтверждает запись.
- **Popup и Alert** — нативные диалоги Telegram (`showPopup`, `showAlert`). Выглядят нативно, не нужно делать свои модалки для простых подтверждений.

**Что лучше не делать:**

- Не использовать тяжёлые фреймворки (Angular, полный React с SSR). Mini App должно грузиться за <1 сек. Лёгкий React/Vue/Svelte или ванильный JS.
- Не пытаться воспроизвести полноценный SPA-роутинг с history API — это конфликтует с навигацией Telegram. Лучше простое переключение экранов (conditional rendering).
- Не запрашивать у пользователя данные, которые уже есть в `initData` (имя, username).
- Не делать горизонтальный скролл — конфликтует с жестами свайпа Telegram.
- Не хранить чувствительные данные в CloudStorage — они не зашифрованы.
- Не полагаться на `window.history` — поведение непредсказуемо в WebView.

**Рекомендуемый стек для MVP:**
- Frontend: React/Vue + Vite (быстрая сборка, малый бандл)
- Стилизация: CSS-переменные Telegram + минимальный CSS
- Данные: JSON-файл или простой API (Supabase / Firebase для MVP)
- Бот: python-telegram-bot или grammY (Node.js)
- Хостинг Mini App: Vercel / Netlify / GitHub Pages (бесплатно)
